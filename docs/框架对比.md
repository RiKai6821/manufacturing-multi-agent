# Agent 开发框架对比（手写 / LangChain / LlamaIndex / AutoGen）

本项目用**多种方式实现了同一套多智能体诊断系统**，目的是吃透每种方式的取舍。
面试核心论点：**先懂底层原理，再用框架；框架是加速器，不是黑盒。**

---

## 一、本项目的实现版本对照

| 能力 | 手写版 | LangChain版 | LlamaIndex版 |
|------|--------|------------|-------------|
| Agent循环 | `main.py` 手写ReAct循环 | `coordinator_langchain.py` create_agent | — |
| RAG检索 | `kb_tools.py` 手写FAISS | — | `kb_tools_llamaindex.py` VectorStoreIndex |
| 工具定义 | 手写JSON schema | `@tool` 装饰器 | 自定义组件 |
| 多智能体 | 协调者-执行者手写 | supervisor + worker | — |

三个版本**功能等价**，可直接对比代码量和控制力。

---

## 二、四种方式的定位

### 1. 手写（原生 API）
```
优点：完全可控——并行、重试、记忆、可观测性、防幻觉都能精确定制
缺点：代码量大，控制流要自己维护
适合：需要深度定制的生产系统（本项目主线选它）
```

### 2. LangChain / LangGraph
```
优点：create_agent 一行搞定ReAct循环，@tool自动生成schema，
      LangGraph的图结构擅长复杂工作流（条件分支、循环、并行节点）
缺点：封装较深，定制需学其扩展点（middleware、checkpointer等）
适合：标准Agent工作流，团队协作（生态成熟）
本项目：coordinator_langchain.py 用 supervisor-worker 重写了协调层
```

### 3. LlamaIndex
```
优点：专注RAG，文档加载/切分/索引/检索全组件化，
      内置多种检索策略（向量/关键词/混合/树形）和后处理器
缺点：Agent能力弱于LangChain，主要强在RAG
适合：以知识库检索为核心的应用
本项目：kb_tools_llamaindex.py 用 VectorStoreIndex 重写了RAG，
       自定义 DashScopeEmbedding 接入百炼（框架扩展点实战）
```

### 4. AutoGen（微软，了解为主）
```
核心思想：把多智能体协作建模为"多个Agent之间的对话"
特色：GroupChat（多Agent群聊）+ GroupChatManager 自动调度发言权
     强调 conversable agent（每个Agent能收发消息、自主回应）
与本项目协调者-执行者的区别：
  本项目：协调者主动"调用"执行者（树状，明确层级）
  AutoGen：Agent之间"对话"，由Manager决定下一个发言者（更对等、更灵活）
适合：需要Agent间自由讨论/辩论的开放式任务
```

AutoGen 代码骨架（概念示意）：
```python
from autogen_agentchat.agents import AssistantAgent
from autogen_agentchat.teams import RoundRobinGroupChat

data_agent = AssistantAgent("数据分析", model_client=..., tools=[查良率, 查报警])
kb_agent   = AssistantAgent("知识检索", model_client=..., tools=[检索知识库])
action_agent = AssistantAgent("行动执行", model_client=..., tools=[建工单])

# 多Agent群聊，Manager自动调度谁发言
team = RoundRobinGroupChat([data_agent, kb_agent, action_agent])
result = await team.run(task="EQP-03良率掉到88%，分析原因并处理")
```

## 三、选型决策树（面试可背）

```
要不要用框架？
├─ 需要深度定制控制流（并行/重试/自定义记忆）→ 手写 或 LangGraph
├─ 标准Agent工作流，要快速搭 → LangChain
├─ 核心是知识库RAG → LlamaIndex
└─ Agent间要自由对话/协商 → AutoGen

向量库选型？
├─ 原型/小数据 → FAISS（内存，快）
└─ 生产/大数据/要持久化过滤 → Milvus

知识 vs 能力 vs 约束？
├─ 知识（会变、要溯源）→ RAG
├─ 能力/风格（固化范式）→ 微调（LoRA）
└─ 约束（规则）→ Prompt工程
```

## 四、一句话面试总结

> "我用原生API手写过完整的多智能体系统（协调者-执行者+并行+记忆+可观测性+防幻觉），
> 也用LangChain的supervisor-worker、LlamaIndex的VectorStoreIndex分别重写过协调层和RAG层，
> 还了解AutoGen的对话式多智能体范式。
> 我的原则是先懂底层再用框架——这样框架出问题时我知道根因在哪，
> 也能根据场景选最合适的工具，而不是被某个框架绑死。"
