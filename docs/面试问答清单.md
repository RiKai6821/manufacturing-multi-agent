按面试官提问频率排序。每题给【考点】+【标准答案】+【加分项】。
核心原则：**用项目里的真实设计和数据回答，不空谈概念。**

---

## 一、Agent 核心（必问）

### Q1：什么是 Agent？它和普通的大模型调用有什么区别？
**考点**：对 Agent 本质的理解
**标准答案**：
> Agent 的本质是一个循环：把问题和可用工具发给模型，模型决定调用工具还是直接回答；
> 调工具就执行并把结果塞回对话，再让模型决定下一步，直到它认为信息足够给出答案。
> 普通大模型调用是一问一答、无法获取外部信息；Agent 能自主规划、调用工具、多步完成复杂任务。

**加分项**：我在项目里既手写过这个循环（main.py 的 `_run_agent`），也用 LangGraph 的 `create_agent` 实现过，所以清楚框架封装的就是这个循环。

---

### Q2：你的多智能体架构是怎么设计的？为什么这么分？
**考点**：架构设计能力
**标准答案**：
> 用「协调者-执行者」架构。一个协调 Agent 负责规划和调度，6 个执行 Agent 各管一个领域：
> 数据分析、知识检索、质量评审、行动执行、保养规划，以及数据探索（动态生成只读SQL，兜住没写死工具的长尾查询）。
> 分的依据是**业务职责边界**——每个 Agent 对应一个职能角色，就像一个诊断团队。
> 协调者把执行 Agent 当工具调用，每个执行 Agent 内部又是一个带专属工具的小循环（子任务自治）。

**加分项**：执行 Agent 我让它"只陈述数据事实、不下最终诊断"，把判断权收归协调者，避免越权——这是用 Prompt 做职责约束。

---

### Q3：Agent 的记忆是怎么实现的？
**考点**：记忆管理（JD 明确要求）
**标准答案**：
> 大模型 API 是无状态的，记忆要自己管。我实现了三层：
> 1. **会话记忆**：一次多轮对话的上下文，带滑动窗口防止 context 溢出，支持追问；
> 2. **诊断历史持久化**：每次诊断结果存入独立的 memory.db，自动抽取症状关键词；
> 3. **经验召回**：新诊断开始前，自动调取该设备/相似症状的历史经验注入上下文，
>    让系统"记得"这台设备以前出过什么问题——形成自我进化的闭环。

**加分项**：我把 memory.db 和代表外部 MES 的 factory.db 物理分开——factory.db 是"外部系统数据"，memory.db 是"Agent 自身状态"，职责不同所以分库。

## 二、RAG（必问）

### Q4：完整讲一下 RAG 的原理和你的实现。
**考点**：RAG 原理
**标准答案**：
> RAG 分两阶段：
> **建库**：文档 → 切分 chunk → embedding 向量化 → 存入向量库；
> **查询**：问题 → 向量化 → 检索最相似的 chunk → 拼进 prompt → 模型看着资料回答。
> 它解决大模型不知道企业内部知识、且容易幻觉的问题。

**加分项**：我做了三件检索优化：① chunk_size/overlap 调参（评测验证200/50最优）；
② 查询加指令前缀提升 BGE 模型准确率；③ 置信度过滤 + 按来源去重，不返回低质量结果。
我还做了评测框架量化效果（Hit Rate、MRR）。

---

### Q5：FAISS 和 Milvus 有什么区别？什么时候用哪个？
**考点**：向量库选型
**标准答案**：
> FAISS 是向量检索库——内存索引、快、但不持久化、不支持过滤，适合原型；
> Milvus 是完整的向量数据库——数据持久化、支持增量更新、标量过滤、分布式，适合生产。
> 原理一致（都是向量相似度搜索），Milvus 在 FAISS 内核上加了工程化外壳。

**加分项**：我项目里两套都实现了（kb_tools.py / kb_tools_milvus.py），还用 LlamaIndex 实现了第三套，切换只改一处 import。选型原则：数据量小/原型用 FAISS，要持久化/过滤/上量用 Milvus。

---

### Q6：怎么防止 RAG/Agent 产生幻觉？
**考点**：输出准确性（JD 要求）
**标准答案**：
> 我做了双层防护：
> 1. **事前约束**：在 system prompt 里强制"只基于工具返回的真实数据作答，缺数据就说未获取，绝不臆测"；
> 2. **事后核对**：报告生成后，用正则提取报告里的关键数字（良率/颗粒数/超期天数），
>    和数据库真实值逐项比对，输出"N/M 项一致"，不一致就标记预警。

**加分项**：数字能自动核对，但 SOP 编号这类细节核对器也会单独提示"可能是模型补充的常识，建议人工复核"——区分了能自动验证和需人工的部分。

## 三、工程化（区分高下的关键）

### Q7：你怎么保证这个系统的质量/可靠性？
**考点**：工程素养（很多候选人答不上）
**标准答案**：
> 几个层面：
> 1. **测试**：24 个单元测试覆盖工具层和记忆层，pytest + GitHub Actions CI 自动跑；
> 2. **错误处理**：工具层输入校验 + 异常隔离（单个工具失败不拖垮整个 Agent）+ 错误码体系；
> 3. **重试**：LLM 调用带指数退避重试，应对 API 限流/网络抖动；
> 4. **评估**：建了 Agent 级评估框架，量化任务完成率、根因命中率、响应时效。

**加分项**：我特意把工具层的测试设计成不依赖大模型 API（确定性逻辑），所以 CI 环境能直接跑，不需要 key。

---

### Q8：你怎么监控和优化 Agent 的运行效果？（JD 职责4原话）
**考点**：监控 + 迭代优化
**标准答案**：
> 监控用可观测性模块：全链路记录每个 Agent 调用、每次工具执行、每次 LLM 调用，
> 带时间戳和统计（耗时/调用次数），可导出 JSON。
> 优化是数据驱动的：我用评估框架先测出基线（平均 71.7s），
> 然后做模型路由优化——执行 Agent 用快模型 qwen-flash，协调者用 qwen-plus，
> 把平均响应时效降到 51.6s、降低 28%，同时任务完成率保持 100%。

**加分项**：还加了流式输出把首字延迟降到 6.6s 改善感知体验，和 embedding 查询缓存。整个过程是"测量→优化→用同一套评估验证"的闭环。

### Q9：这个系统怎么部署？怎么保证高可用和扩展性？
**考点**：云原生部署（JD 要求）
**标准答案**：
> 三级演进：本地运行 → Docker 容器化 → Kubernetes 编排。
> Docker 做标准化打包（分层构建、密钥运行时注入、数据卷持久化）；
> K8s 用 Deployment 管理多副本（挂了自愈）、Service 做负载均衡、HPA 按 CPU 自动扩缩容、
> Secret 管理密钥、liveness/readiness 探针做健康检查。

**诚实补充**（如果追问）：配置和方案我都写好了，本地环境所限还没在真实集群跑过，但理解每个组件的作用。

## 四、模型与选型

### Q10：什么时候用微调，什么时候用 RAG，什么时候用 Prompt？
**考点**：技术选型判断（高频）
**标准答案**：
> 三者是配合不是替代：
> - **知识**（会更新、要溯源）→ RAG，比如我的企业知识库走检索；
> - **能力/风格**（固化思维范式和输出格式）→ 微调，比如让模型养成固定诊断范式；
> - **约束**（规则）→ Prompt，比如防幻觉纪律。

**加分项**：我准备了微调数据集——从自己的历史案例库自动生成"症状→根因"样本，
用百炼 LoRA 微调（无需自备 GPU）。LoRA 是冻结原模型只训低秩矩阵，参数量降到1%以下，
训练快、可热插拔，是工业界主流。

---

### Q11：LoRA 的原理是什么？
**考点**：微调技术
**标准答案**：
> 全量微调要更新模型全部参数，慢且贵。LoRA 冻结原模型权重，
> 在旁边插入两个低秩矩阵 A×B 只训练它们，参数量降到原来的 0.1%~1%，
> 显存小、训练快、效果接近全量，而且可以热插拔不同的 LoRA 适配器。
> P-tuning 更轻，只训练连续的提示向量。

---

### Q12：模型 API 调用，messages 里的 role 有哪些？
**考点**：API 基础
**标准答案**：
> system（规则/人设）、user（用户说的）、assistant（模型说的）、tool（工具返回结果）。
> 多轮对话靠每次把完整 history 重新发过去实现，因为 API 是无状态的。

## 五、可能的深挖/压力题

### Q13：你的系统响应要 50 多秒，太慢了，生产能用吗？
**诚实+思路**：
> 当前是诊断类场景，准确性优先于速度，50s 内完成完整的多Agent诊断+核对是可接受的。
> 但我做了优化（模型路由降28%、流式输出改善感知），还能继续优化：
> 工具结果缓存、投机执行、更细的模型路由、prompt缓存。
> 关键是我有评估框架能量化每次优化的效果，而不是凭感觉。

### Q14：如果让你接真实的 MES/ERP 系统，怎么改？
**标准答案**：
> 我的工具层做了接口隔离——db_tools 里每个函数下面都注释了真实接入写法
> （MES 的 REST API、Oracle 直连、SAP OData、IoT 平台的 OPC-UA）。
> 切换只需替换函数内部实现，函数签名和返回格式不变，Agent 层完全无感知。
> 这就是分层架构的价值。

### Q15：你这个项目最大的难点/最有成就感的地方？
**参考**：
> 难点是让多个 Agent 稳定协作且输出可信——不是堆功能，而是加了防幻觉、
> 评估框架、可观测性这些"可信工程"的东西。
> 最有成就感的是建了量化评估体系后，用数据驱动把响应时效降了28%——
> 从"感觉变快了"变成"测出来快了28%"，这是工程师和调包侠的区别。
