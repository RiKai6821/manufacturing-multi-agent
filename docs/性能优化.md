# 响应时效优化

对应 JD 职责4：「监控 Agent 运行效果（任务完成率、**响应时效**）；迭代优化工作流」。

本系统从最初的串行全量 qwen-plus，逐步优化到并行+模型路由+缓存，
**平均响应时效从 ~90s 降到 ~52s**，并有实测数据支撑。

---

## 一、优化手段与实测效果

### 1. 并行执行（已实现，main.py）
协调者在同一轮请求多个无依赖的 Agent 时，用线程池并发跑。
```
数据分析Agent ──┐
                ├── 同时进行（而非串行等待）
知识检索Agent ──┘
```
效果：单次诊断从 ~90s 降到 ~77s。

### 2. 模型路由（已实现，settings + executor_agents）⭐
不同角色用不同模型，平衡速度/成本/质量：
```
协调者（综合判断、出报告）→ qwen-plus（强模型，少量调用）
执行Agent（查数据/检索/建单）→ qwen-flash（快模型，大量调用）
```
**实测对比（eval_agent.py 同一组场景）：**
| 配置 | 平均响应时效 | S1场景 |
|------|------------|--------|
| 全 qwen-plus | 71.7s | 80.9s |
| 执行Agent换 qwen-flash | **51.6s** | 64.3s |
| **改善** | **↓28%** | ↓20% |

任务完成率、根因命中率均保持 100%——说明简单任务用快模型不掉质量。

### 3. Embedding 查询缓存（已实现，kb_tools）
诊断中多个 Agent 常检索相似问题，对查询向量化做 LRU 缓存：
```python
@functools.lru_cache(maxsize=128)
def _embed_query_cached(query): ...
```
效果：重复/相似查询命中缓存，省去 embedding API 网络往返。

### 4. 索引缓存（已实现，kb_tools）
FAISS 索引序列化到本地 pkl，重启不重建，且配置变化自动失效。

### 5. 流式输出（示例，stream_demo.py）⭐
改善"感知延迟"：报告 token 边生成边显示。
**实测：** 首字延迟(TTFT) 6.64s，总耗时 12.11s。
用户 6.64s 就看到内容，而非干等 12s——感知延迟降低约 45%。

## 二、优化前后总览

```
最初版本：串行 + 全量 qwen-plus              ~90s，纯等待
   ↓ 并行执行
   ↓ 模型路由（执行Agent用flash）
   ↓ embedding/索引缓存
   ↓ 流式输出（感知优化）
现在版本：~52s 实际 + 6.6s 首字可见
```

---

## 三、可继续优化的方向（面试可讲）

```
- 工具结果缓存：相同设备短时间内重复查询直接返回缓存
- 投机执行：协调者规划时预启动大概率要调的Agent
- 批量embedding：多个查询合并一次API调用
- 更细的模型路由：用小模型做"是否需要工具"的预判，减少大模型轮次
- KV-Cache/Prompt缓存：固定的system提示走平台的prompt缓存
```

---

## 四、一句话面试总结

> "我用 eval 框架量化了响应时效，通过模型路由（执行Agent换轻量模型）把平均诊断
> 时延从71.7s降到51.6s、降低28%，且任务完成率保持100%；再用流式输出把首字延迟
> 降到6.6s改善感知体验。优化都是数据驱动的——先测量、再优化、用同一套eval验证效果。"
